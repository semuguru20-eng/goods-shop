# 굿즈샵 디자인 통일 스킬

이 스킬은 `goods-shop`의 모든 페이지가 같은 디자인 언어를 쓰도록 만드는 규칙이다.
페이지를 새로 만들거나 기존 페이지를 수정할 때마다 아래 규칙을 확인하고 지킨다.

## 0. 기본 원칙: 새로 만들지 말고 재사용한다

색상, 버튼, 카드, 입력창 스타일은 전부 `css/style.css`에 이미 정의되어 있다.
새 페이지나 컴포넌트를 만들 때 **인라인 스타일이나 새로운 hex 색상 값을 만들지 말고**,
아래에 정리된 기존 CSS 변수/클래스를 그대로 가져다 쓴다. 새로운 스타일이 꼭 필요하면
`css/style.css`에 추가하고 모든 페이지가 같은 파일을 공유하게 한다.

**새 페이지는 이 스킬 폴더의 `signup.html`을 기반으로 시작할 것.** 공통 헤더(로고 /
메뉴 / 로그인 상태)와 공통 푸터(회사 정보 / 약관 링크)가 이미 갖춰져 있고 본문
영역만 비어 있으니, 새 페이지를 만들 때는 이 파일을 프로젝트 루트로 복사해 파일명과
`<title>`을 바꾸고 `<main>` 안에 내용을 채운다. 헤더/푸터 구조를 처음부터 새로 만들지
않는다.

## 1. 색상 — 항상 같은 CSS 변수 사용

`css/style.css`의 `:root`에 정의된 변수만 사용한다. 페이지마다 색을 새로 정의하지 않는다.

| 변수 | 값 | 용도 |
|---|---|---|
| `--bg` | `#0f0f14` | 페이지 배경 |
| `--card` | `#1b1b24` | 카드/패널 배경 |
| `--border` | `#2a2a36` | 테두리 |
| `--text` | `#eceef3` | 기본 글자색 |
| `--text-dim` | `#9a9aab` | 보조 설명 글자색 |
| `--accent` | `#4d7fd9` | 강조색 (주요 버튼, 가격, 링크 hover) |
| `--danger` | `#d9534d` | 에러/삭제/실패 |

상태 색상도 통일: 성공 `#4caf50`(`.status-paid`), 대기 `#e0a840`(`.status-pending`),
실패는 `--danger`(`.status-failed`, `.error`).

## 2. 버튼 — 모양과 크기를 통일

- **모든 페이지의 버튼 모서리는 부드럽게 둥글게 만든다.** 각진(0px) 모서리는 쓰지
  않는다. `button.primary`, `.tabs button` 모두 이미 `border-radius: 6px`가 적용돼
  있으니 그대로 재사용하고, 새 버튼 클래스를 만들 때도 반드시 `border-radius: 6px`
  이상을 준다.
- 주요 액션 버튼(구매하기, 로그인, 회원가입 등)은 항상 `button.primary` 클래스를 쓴다.
  (`background: var(--accent)`, `color: #fff`, `border-radius: 6px`)
- 탭/토글형 버튼은 `.tabs button` 패턴을 재사용한다.
- **터치하기 쉬운 크기 유지**: 버튼의 세로 높이(패딩 포함)가 최소 44px 이상이 되도록 한다.
  `button.primary`의 기존 `padding: 10px 16px`은 폰트 크기를 키우면(아래 4번) 자연히
  44px 근처가 되므로, 폰트 크기를 줄여서 버튼을 작게 만들지 않는다.
- 버튼 사이 간격은 최소 8px 이상 띄워서 옆 버튼과 오터치가 나지 않게 한다.
- 위험한 동작(삭제, 취소 등) 버튼은 `--danger` 색을 쓰고, 일반 버튼과 시각적으로
  구분되게 한다.

**[좋은 버튼 예시]**
- 손가락으로 편하게 누를 만큼 크게 (→ 위 44px 이상 규칙)
- 모서리는 부드럽게 둥근 모양 (→ `border-radius: 6px`)
- 색은 화면에서 가장 잘 보이게 (→ `var(--accent)` 배경 + 흰 글자)

이 세 가지가 새 버튼을 만들 때의 기준이다. 모든 페이지의 버튼을 이 예시에 맞춰 만든다.

## 3. 입력칸 — 라벨과 에러 표시

**[좋은 입력칸 예시]**
- 입력칸 위에 무엇을 적는 칸인지 라벨 표시
- 잘못 입력하면 빨간색으로 알려주기

- 입력칸마다 위에 `<label>`을 붙여서 무엇을 입력하는 칸인지 알 수 있게 한다.
  placeholder만으로 대체하지 않는다 (placeholder는 값을 입력하면 사라져서 라벨 역할을
  하지 못한다).
- 값이 잘못됐을 때는 `--danger` 색으로 테두리와 안내 문구를 보여준다. 안내 문구는
  "6. 문구" 규칙(해요체)을 따른다.
- 이 예시가 새 입력칸을 만들 때의 기준이다. 앞으로 만드는 모든 입력 폼(로그인,
  회원가입 등)을 이 기준에 맞춰 만든다.

## 4. 글자 크기 — 모바일에서도 잘 보이게

- 본문 기본 글자 크기는 최소 16px(`1rem`) 이상을 유지한다. 이보다 작게 줄이지 않는다.
- 보조 설명 텍스트(`.text-dim`, `.nav-email` 등)도 최소 14px(`0.9rem`) 밑으로 내려가지
  않게 한다.
- 제목(`h1`)은 페이지 성격에 따라 `1.4rem`~`1.8rem` 범위를 유지해서 다른 페이지와
  위계가 어긋나지 않게 한다 (`.hero-info h1`이 `1.8rem`인 것을 기준으로 삼는다).
- 폰트 패밀리는 항상 `body`에 지정된 시스템 폰트 스택(`-apple-system, "Segoe UI",
  "Malgun Gothic", sans-serif`)을 그대로 쓴다. 페이지별로 다른 폰트를 지정하지 않는다.

## 5. 반응형 — 모바일에서 자연스럽게

- 모든 HTML 문서 `<head>`에 `<meta name="viewport" content="width=device-width,
  initial-scale=1.0">`가 있는지 확인한다. 새 페이지를 만들 때 빠뜨리기 쉬운 부분이다.
- 목록/그리드는 `.product-grid`처럼 `grid-template-columns: repeat(auto-fill,
  minmax(220px, 1fr))` 방식을 써서 화면 너비에 따라 자동으로 열 개수가 줄어들게 한다.
  고정 컬럼 수(`repeat(3, ...)` 등)를 쓰지 않는다.
- 좌우로 나열되는 레이아웃(`.hero`처럼)은 `flex-wrap: wrap`을 걸어서 좁은 화면에서
  세로로 자연스럽게 쌓이게 한다.
- 본문 컨테이너는 `main`처럼 `max-width`를 잡고 좌우 `padding`을 둬서, 큰 화면에서는
  너무 안 늘어나고 작은 화면에서는 화면 끝에 글자가 붙지 않게 한다.
- 표(`table`)처럼 좁은 화면에서 깨지기 쉬운 요소는 가로 스크롤이 필요한지 확인하고,
  필요하면 감싸는 요소에 `overflow-x: auto`를 준다.

## 6. 문구 — 같은 의미는 같은 표현으로

사용자에게 보여주는 문구는 아래 톤·패턴을 따르고, 이미 쓰인 문구가 있으면 그 표현을
그대로 재사용한다 (새로 비슷한 말을 만들지 않는다).

**말투 규칙**
- 해요체(정중체)로 통일: "~합니다", "~해주세요", "~할까요?"
- 확인이 필요한 위험한 동작은 항상 물음표로 끝나는 확인 문구를 쓴다.
  예: "삭제할까요?", "정말 취소할까요?" (이후 삭제 관련 기능을 추가할 때도 이 패턴을
  그대로 따른다 — "삭제하시겠습니까?", "정말요?" 같은 변형을 섞어 쓰지 않는다)

**현재 사용 중인 기준 문구 (그대로 재사용)**
- 접근 제한: "관리자만 접근할 수 있는 페이지입니다."
- 로딩/조회 실패: "OO을(를) 불러오지 못했습니다." (예: "상품을 불러오지 못했습니다")
- 잘못된 접근: "잘못된 접근입니다. 상품 목록에서 다시 시도해주세요."
- 처리 실패: "OO 실패: " + 에러 메시지 (예: "주문 생성 실패: ...")
- 완료 안내: "OO 완료! ..." (예: "회원가입 완료! 자동으로 로그인됩니다.")
- 결제 결과: "결제가 완료되었습니다!" / "결제 승인에 실패했습니다."
- 로그인 필요: "로그인이 필요합니다."

새 문구를 추가해야 하면 위 패턴(대상 + 상태 + 해요체)을 따르고, 이 목록에도 함께
추가해서 다음에 같은 상황이 생기면 재사용하게 한다.

## 7. 금지 규칙

아래 네 가지는 절대 하지 않는다.

- **각진 버튼 금지**: `border-radius: 0`인 버튼, 즉 모서리가 각진 버튼은 어떤
  페이지에도 쓰지 않는다. 항상 2번(버튼) 규칙대로 `border-radius: 6px` 이상의
  둥근 모서리를 쓴다.

- **이모지·아이콘 금지**: 버튼, 문구, 상태 표시 어디에도 이모지나 아이콘을 쓰지 않는다.
  글자로만 의미를 전달한다 (예: "완료 이모지 + 완료"가 아니라 "완료"만, "장바구니
  아이콘 + 구매하기"가 아니라 "구매하기"만 쓴다).
- **페이지마다 다른 색 금지**: 1번(색상) 표에 없는 색은 어떤 페이지에서도 쓰지 않는다.
  같은 용도(배경/버튼/에러 등)는 모든 페이지에서 항상 같은 변수를 쓴다.
- **같은 의미, 다른 단어 금지**: 같은 동작을 가리키는 단어를 페이지마다 다르게 쓰지
  않는다 (예: "저장" / "확인" / "완료"를 섞어 쓰지 말고, 하나를 정해서 모든 페이지에서
  그 단어만 쓴다). 새 단어를 정해야 하면 6번(문구) 목록에 추가하고 그 뒤로는 그 단어만
  재사용한다.

## 체크리스트 (새 페이지/컴포넌트 작업 시)

- [ ] `css/style.css`만 불러오고, 새 색상 hex 값을 인라인으로 넣지 않았는가
- [ ] 주요 버튼에 `button.primary`(또는 기존 버튼 클래스)를 썼는가
- [ ] 본문 글자 16px 이상, 버튼 터치 영역 44px 이상을 유지했는가
- [ ] `<meta name="viewport">`가 있는가, 그리드/flex가 좁은 화면에서 자연스럽게
      줄바꿈되는가
- [ ] 사용자에게 보여줄 문구가 기존 목록에 있으면 그대로 썼는가, 새로 만들었으면
      이 문서의 "6. 문구" 목록에 추가했는가
- [ ] 입력칸마다 라벨이 있는가, 잘못 입력했을 때 빨간색으로 알려주는가
- [ ] 이모지·아이콘을 쓰지 않고 글자로만 표시했는가
- [ ] 모든 버튼의 모서리가 둥근가 (`border-radius: 6px` 이상, 각진 버튼 없음)
- [ ] 1번(색상) 표에 없는 색을 새로 쓰지 않았는가
- [ ] 같은 의미의 동작에 다른 단어(저장/확인/완료 등)를 섞어 쓰지 않았는가
